## 一、产品定位

| 项目 | 内容 |
|------|------|
| 本质 | 代码探伤仪（不是编辑器、不是调试器）|
| 核心价值 | 把「人工逐行分析屎山代码」变成「AI识别 + 可视化呈现」|
| 目标用户 | 接手非自己写的屎山代码的程序员 |
| 使用场景 | 接手新项目 / 代码审查 / 重构前分析 / 技术债评估 |
| 产品比喻 | 人类看病：体检扫描 → 报告可视化 → 定位病灶 → 修复 → 巩固复查 |

## 二、核心工作流程（水龙头比喻）

```
【第1步】用户先切换到「探伤模式」
      ↓
【第2步】检查水龙头状态，决定显示内容
      ↓
┌─────────────────────────────────────────────────┐
│  状态A：没水（未接入代码）                    │
│  → 播放演示视频（固定脚本）                   │
│  → 展示「探伤功能能干嘛」                    │
└─────────────────────────────────────────────────┘
         ↓ 用户接入代码（来水了）
┌─────────────────────────────────────────────────┐
│  状态B：有水但没开水龙头（已接入、未扫描）  │
│  → 依然播放演示视频                          │
│  → 提示用户「代码已接入，请点击开始扫描」  │
└─────────────────────────────────────────────────┘
         ↓ 用户点击「开始扫描」（开水龙头）
┌─────────────────────────────────────────────────┐
│  状态C：正式探伤（最核心部分）               │
│  → 深度扫描代码                              │
│  → 呈现「代码体检报告」可视化              │
│  → 支持反向定位（点击节点 → 跳转真实代码行）│
│  → 针对问题代码进行修复/优化/改良          │
└─────────────────────────────────────────────────┘
         ↓ 用户点「结束」或关闭页面
┌─────────────────────────────────────────────────┐
│  状态D：演示模式（等待下一次接入）           │
│  → 自动回到演示模式                        │
└─────────────────────────────────────────────────┘
```

### 状态转换规则

| 转换 | 触发条件 | 目标状态 |
|--------|---------|---------|
| 任意状态 → 状态A | 未接入代码（或用户主动切回演示） | 演示模式 |
| 状态A → 状态B | 用户上传代码（来水了） | 演示模式（仍演示）|
| 状态B → 状态C | 用户点「开始扫描」 | 正式探伤 |
| 状态C → 状态B | 用户点「取消扫描」 | 演示模式 |
| 状态C → 状态A | 扫描完成 + 用户点「结束」 | 演示模式 |
| 演示中打断 | 用户接入代码或点「开始扫描」 | 立即跳演示 |

## 三、功能模块

### M1：模式切换

- 入口：右上角模式按钮（观察/调试/实景 → 新增「探伤」）
- 切换后：保留现有 10 节点结构，只改变显示内容
- 切换动画：有过渡动画（节点颜色渐变过渡）

### M2：演示模式（状态A、B通用）

- **内容**：固定脚本，内置屎山代码示例（故意写好的有问题代码）
- **流程**：模拟导入 → 扫描 → 可视化（问题节点变颜色）→ 修复（动画假装修复）→ 树恢复全绿
- **时长**：30-60 秒（中）
- **文字提示**：不要
- **语音旁白**：不要
- **可跳过**：能，有信号（用户操作）就来演示，进入当前模式
- **随机映射**：每次演示随机挑 2-8 个节点变红（避免用户看腻）
- **演示代码位置**：单独放 `demo-code.js` 文件（独立，好维护）

### M3：代码接入

- **方式**：上传文件夹（`<input webkitdirectory>`）
- **支持**：Chrome/Edge/Firefox 都支持
- **检查**：上传后、扫描前，自动检查文件结构是否符合规范
  - 符合 → 允许开始扫描
  - 不符合 → 弹窗提示「检测到不符合要求的程序，请规范后再尝试提交」
  - 弹窗后等待 1 分钟，没有新操作自动回到演示模式
  - 等待中途用户上传新代码 → 立即取消等待，处理新代码
- **规范标准**：必须符合 acorn 能解析的 JS/TS 代码（最小运行标准，能达到一般小程序/小屎山运行效果即可）
- **非 JS/TS 代码**：MVP 只支持 JS/TS，其他语言后续版本再加（弹窗提示「目前只支持 JS/TS」）

### M4：代码扫描

- **工具**：acorn（JS/TS 静态分析）
- **运行位置**：浏览器里做（用 Web Worker，不卡界面）
- **方式**：渐进式扫描（边扫边呈现，不限制代码大小）
- **进度显示**：顶部进度条（类似 YouTube 加载视频的红色进度条）
- **可取消**：要，扫描过程中显示「取消扫描」按钮，点了立即停止，回到演示模式
- **语法错误处理**：弹窗让用户选「自动修复并重新扫描」或「取消，我自己重新上传」
- **扫描前检查**：如果代码有语法错误，acorn 解析失败 → 走语法错误处理流程

### M5：体检报告可视化

- **主要呈现**：扶桑树节点变颜色（问题节点用质量颜色编码）
- **扫描完成后**：先显示「扫描完成」弹窗，等用户点「查看报告」才进入正式探伤界面
- **信息层次**：
  1. 概览（一眼看出全局）：所有节点用颜色编码，问题节点有轻微脉冲动画，右上角显示「代码健康评分」
  2. 聚焦（点击节点后）：当前节点高亮，其他节点淡化，InfoCard 显示问题详情
  3. 传播路径（可选）：点击节点后，高亮「问题传播路径」（相关连线变红+脉冲动画）
  4. 对比（修复前后）：修复后重新扫描，节点颜色动态过渡（红→黄→绿）
- **节点问题展示**：
  - 单个问题：点击节点 → 直接跳转到问题代码
  - 多个问题：节点角标显示问题数量（如「3!」），点击后展开弹框，按严重程度排序显示所有问题，逐一点击跳转到对应问题代码

### M6：反向定位

- **内部查看**：用 Monaco Editor（VS Code 同款，开源 MIT 协议）
  - 点击问题 → 在 InfoCard 下方展开代码片段（上下文 10 行）
  - 用户可以「在内嵌编辑器里直接修改」
- **外部跳转**：MVP 外接限定 VS Code（最主流编辑器，~85% JS/TS 开发者使用）
  - 具体方式：只做「复制文件路径 + 行号」，用户手动去 VS Code 找（最简单，1 小时内完成）
  - 后续版本再开发 VS Code 插件，实现「一键跳转」
- **跳转后代码行高亮**：整行高亮 + 闪烁效果

### M7：修复验证

- **修复方式**：用户根据跳转的代码行，手动修复（MVP 不做 AI 自动修复）
- **重新扫描**：
  - 手动点「重新扫描」按钮
  - 全部修复后（所有节点变绿），自动弹窗提示「是否重新全面扫描（病情巩固）」
  - 用户点「是」→ 跳到扫描界面，要再点一次「开始扫描」（半自动）
- **迭代修复**：直到整棵树全绿 + 再次扫描全绿 = 完全修复成功
- **修复成功**：弹窗「修复成功！是否结束？」→ 用户点「结束」→ 回到演示模式

### M8：报告导出

- **格式**：Markdown（方便程序员阅读和编辑）
- **类型**：
  1. 探伤扫描报告（扫描完成后导出）
  2. 修复建议报告（AI 生成的修复建议，MVP 不做，后续版本）
  3. 修复对比报告（修复前后对比，MVP 不做，后续版本）
- **按钮位置**：底部状态栏
- **主动索取**：不是强制展示，用户需要才点（不需要就不用点）

## 四、技术架构

| 层级 | 技术选型 | 说明 |
|-------|---------|------|
| 前端渲染 | 现有 Canvas 2D | 零外部依赖，继续使用 |
| 静态分析 | acorn（浏览器版本 + Web Worker） | 不卡界面，代码不离开本地 |
| 代码跳转 | Monaco Editor（内部）+ 复制路径（外部 VS Code） | MVP 内部用 Monaco，外部只复制路径 |
| 数据模型 | tasks.json 扩展 quality 对象 | 现有 10 节点 + quality 字段 |
| 节点映射 | 固定规则（MVP）| 基于 CommonJS/ES6 项目常见结构写死映射规则 |
| 演示数据 | 同一套节点数据（现有 10 节点 + quality 字段） | 演示和真实扫描用同一套，不分开 |
| 状态保存 | 不保存（MVP）| 页面关闭后不保存扫描结果，重新打开要重新上传、重新扫描 |
| 放大/缩小 | 支持（现有 Canvas 已有缩放逻辑，暴露给用户） | 鼠标滚轮放大缩小 |
| i18n | 全覆盖（中英文切换） | 所有新增 UI 文字都加进 i18n 对象 |

### 固定映射规则（MVP）

基于 CommonJS/ES6 项目常见结构：

| 扶桑树节点 | 对应文件/目录 |
|------------|-------------|
| entry | main.js / index.js / app.js |
| config | config/ / .env / webpack.config.js |
| db | models/ / database/ / db.js |
| cache | cache/ / redis.js |
| api | api/ / routes/ / controllers/ |
| auth | auth/ / middleware/ / passport.js |
| frontend | components/ / pages/ / App.vue |
| tests | test/ / __tests__/ / *.test.js |
| ci | .github/ / Jenkinsfile / Dockerfile |
| goal | README.md / package.json |

**不符合规范的处理**：扫描前提示，不符合就弹窗「请调整结构，或等待后续版本」。

| 问题类型 | 颜色 | 心理学依据 | 紧急程度表达 |

| 循环依赖 | 🟣 紫色 | 复杂、异常、需要仔细分析 | 高饱和 + 旋转光环 |

| 缺少错误处理 | 🔵 蓝色 | 需要行动、信任风险 | 中饱和 + 常亮边框 |

### 动画特效（交互时触发，同一时间最多 1-2 个节点播放）

| 问题类型 | 动画 | 触发时机 |
|---------|------|---------|
| 循环依赖 | 紫色漩涡旋转 | 点击选中 / 演示模式 |

| 死代码 | 灰色呼吸闪烁 | 演示模式高亮时 |
| 超大函数 | 节点膨胀收缩 | 点击选中 |
| 缺少错误处理 | 蓝色常亮脉冲 | 演示模式 |

### 信息层次

1. **概览**（一眼看出全局）：节点颜色 + 轻微脉冲 + 健康评分
2. **聚焦**（点击节点后）：当前节点高亮 + 其他节点淡化 + InfoCard 详情
3. **传播路径**（可选）：高亮问题传播路径（连线变红 + 脉冲）

## 六、交互设计

### 模式切换交互

- 入口：右上角模式按钮
- 切换后：现有树保留，只改变显示内容
- 能从「探伤模式」切回其他模式（随时可切）

### 节点交互

- **鼠标悬停**：节点轻微脉冲（1.0→1.05 缩放），显示问题简介（tooltip）
- **单击（单个问题）**：直接跳转到问题代码
- **单击（多个问题）**：展开弹框，显示所有问题（按严重程度排序），逐一点击跳转到对应问题代码
- **双击**：暂无定义（后续版本可定义「快速修复」）

### 代码接入交互

- 点击「上传文件夹」按钮 → 弹出文件夹选择器 → 自动读取所有文件
- 上传后，自动检查文件结构是否符合规范 → 符合则允许扫描，不符合则弹窗提示

### 扫描交互

- 点击「开始扫描」→ 顶部进度条显示扫描进度（当前正在扫描哪个文件）
- 扫描过程中，显示「取消扫描」按钮 → 点了立即停止，回到演示模式
- 扫描完成后，弹窗提示「扫描完成，是否查看报告？」→ 用户点「查看报告」才进入正式探伤界面

## 七、数据流

```
代码接入 → 存储（浏览器内存）→ 触发 UI 更新
    ↓
扫描 → 生成 quality 数据（acorn 分析结果）→ 更新 tasks.json → 重绘扶桑树
    ↓
反向定位 → 跳转代码（Monaco Editor 内嵌 / 复制路径给 VS Code）
    ↓
修复 → 重新扫描 → 更新可视化（节点颜色动态过渡）
    ↓
修复成功（全绿 + 再次扫描全绿）→ 弹窗提示 → 用户点「结束」→ 回到演示模式
```

## 八、异常场景处理

| 异常场景 | 处理方式 |
|---------|---------|
| 接入的代码有语法错误（acorn 解析失败）| 弹窗让用户选「自动修复并重新扫描」或「取消，我自己重新上传」 |
| 代码太大，扫描超时 | 渐进式扫描，边扫边呈现，不限制大小 |
| 浏览器内存不足 | 不处理（MVP），用户刷新页面重新扫描 |
| 静态分析工具崩溃 | 弹窗提示「扫描失败，请重新尝试」|
| 用户网络断开（如果用后端扫描）| MVP 不做后端扫描，不存在此问题 |
| 用户不小心关闭了页面 | 不保存扫描结果，重新打开要重新上传、重新扫描 |
| 非 JS/TS 代码 | 弹窗提示「目前只支持 JS/TS，请重新上传」|

## 十、验收标准（MVP）

### 核心流程验收

1. ✅ 切换「探伤模式」→ 播放演示脚本（30-60 秒，无文字/语音，可跳过）
2. ✅ 上传文件夹 → 自动检查结构是否符合规范 → 符合则允许扫描
3. ✅ 点击「开始扫描」→ 顶部进度条显示进度 → 可取消
4. ✅ 扫描完成 → 弹窗提示 → 点「查看报告」→ 扶桑树节点变颜色
5. ✅ 点击问题节点 → 内部 Monaco Editor 展开代码片段 → 可手动修改
6. ✅ 点击「复制路径」→ 复制文件路径 + 行号 → 用户手动去 VS Code 找
7. ✅ 修复代码后 → 点「重新扫描」→ 节点颜色动态过渡（红→黄→绿）
8. ✅ 全绿后 → 自动提示「是否重新全面扫描」→ 点「是」→ 再次扫描全绿 → 弹窗「修复成功！是否结束？」
9. ✅ 点「结束」→ 回到演示模式
10. ✅ 报告导出 → 底部状态栏按钮 → 点击导出 Markdown 报告

### 边界情况验收

1. ✅ 上传非 JS/TS 代码 → 弹窗提示「目前只支持 JS/TS」
2. ✅ 上传代码有语法错误 → 弹窗让用户选「自动修复」或「取消」
